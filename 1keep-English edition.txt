API, which stands for Application Programming Interface, can be understood as a translator and communication channel between two different systems. For example, to enable a mobile graphics card to work for a game, graphics API is needed to relay rendering instructions to the graphics card. Next, we will discuss three core graphics APIs. The first is OpenGL, the most commonly used open-source graphics specification on desktop computers, officially known as the Open Graphics Library. It is specifically used to send rendering instructions for 3D/2D games, allowing the graphics card to render the game visuals. Its core operating logic is a global implicit state machine model. You can think of the state machine as a control panel with a single master switch; all modifications to the visuals will alter this master switch, and all subsequent drawing operations will fully depend on the current settings of that switch. OpenGL will automatically handle instruction packaging, state merging, and synchronization operations for you without any manual intervention. The second is OpenGLES, a simplified version of OpenGL specifically designed for mobile devices like phones and tablets, officially called OpenGL for Embedded Systems. Its core principle is the same as OpenGL, also being a driver-dominant implicit state machine model, but it removes many redundant features from desktop OpenGL that are unnecessary for mobile use, retaining only the core capabilities of the programmable pipeline. This significantly reduces the operational overhead for the graphics card driver while enforcing precision qualifiers for shaders to adapt to the computational limitations of mobile GPUs. It also features optimizations specifically for the Tile-Based Rendering (TBR) architecture of mobile GPUs. TBR means that mobile GPUs do not render a complete frame all at once but break the frame into many small tiles to be rendered separately, greatly reducing memory bandwidth usage and power consumption. The third is Vulkan, a new generation open-source graphics specification that is completely opposite to OpenGL. It employs an explicit rendering model without global state, which means that all the work previously done automatically by the OpenGL driver is now entirely under your manual control. The driver only translates the instructions you provide into machine code that the graphics card can understand, without performing any extra implicit optimizations or state management. Its core advantage is extremely low CPU overhead, support for multithreaded parallel recording of rendering commands, allowing full use of mobile multi-core CPU performance, and enabling precise control over graphics memory allocation, queue scheduling, and synchronization timing, thus completely resolving compatibility issues arising from different mobile manufacturers' drivers. Next, we will discuss the most essential differences between these three APIs. OpenGL and OpenGLES are considered driver-dominated implicit state machine models, where all rendering details are managed by the driver, which can lead to high CPU overhead and single-thread bottlenecks. In contrast, Vulkan is a developer-dominated explicit stateless pipeline model, where all rendering details are manually controlled, resulting in extremely low CPU overhead and full hardware performance release, but it comes with a high development threshold. Now, let's discuss renderers. A renderer is a full-link translation adaptation layer between the game’s native rendering instructions and the underlying graphics API and graphics card driver. Since the Java version of Minecraft natively uses desktop OpenGL instructions while mobile only supports OpenGLES and Vulkan instructions, the core function of the renderer is to fully receive the desktop OpenGL instructions sent by the game, translate them into instructions recognizable by mobile graphics cards, and then pass them to the driver and graphics card for execution. Next, I will discuss the underlying principles of each renderer one by one. The first is the MobileGlues renderer, which is an open-source OpenGL compatibility layer specifically designed for the mobile version of Minecraft.3 The specification's instructions are converted into OpenGLES or Vulkan commands supported by mobile devices. Its architecture is divided into four main modules: the front-end API implementation layer, the state tracking management layer, the back-end translation layer, and the auxiliary tool module. The front-end API implementation layer is responsible for receiving all OpenGL calls from the game and verifying the validity of the parameters. The state tracking management layer is its core data hub, which records all rendering-related settings in real-time. Changes are only passed to the graphics driver when they truly occur, avoiding redundant operations that waste performance. The back-end translation layer supports two modes: one converts OpenGL commands directly into OpenGLES commands, and the other converts them into Vulkan commands, automatically switching based on the mobile hardware. It also has several core optimization mechanisms. The first is the lazy binding mechanism, which does not immediately send rendering setting changes to the GPU but waits until the actual frame rendering to package and submit all changes to the GPU at once, significantly reducing CPU overhead from frequent setting switches. The second is the multi-threaded rendering architecture, which offloads non-urgent tasks like preparing rendering data and compiling shaders to background threads, while the main thread is only responsible for sending draw commands to the GPU, fully utilizing the mobile device's multi-core CPU and avoiding main thread stuttering. The third is the shader caching mechanism, which caches the translated and compiled shader codes on the mobile device, so that when reusing the same lighting shader, it can be read directly from the cache without re-translating or compiling, greatly shortening lighting loading time. The fourth is an incremental state update strategy, which only sends the actually changed rendering settings to the GPU, avoiding redundant communication overhead. The second is the GL4ES renderer, an open-source OpenGL compatibility layer that corely converts desktop OpenGL 1.x/2.x commands into OpenGLES 1.1/2.0 commands supported by mobile devices. It runs on the foundation of an API interception mechanism, encapsulating function interfaces that are identical to those of desktop OpenGL, intercepting requests directly when the game calls OpenGL functions, and converting them into OpenGLES commands executable on mobile devices without modifying any game code. The third is the ANGLE renderer, officially known as Almost Native Graphics Layer Engine, which is an open-source cross-platform graphics middleware developed under Google's leadership. Its core is the non-intrusive conversion of the full standard API of OpenGLES to the graphics API natively supported by the target platform, completely solving the compatibility issues caused by the varying quality of OpenGLES drivers from different smartphone manufacturers. Next, let’s discuss the underlying principles and terminology related to the Android system. First, let's talk about the Android SDK version. SDK stands for Software Development Kit. The Android SDK version corresponds to different versions of the Android system. Different API levels of the SDK will have different system call interfaces and permission control policies. The underlying principle is that different SDK versions will modify or truncate interactions with the underlying kernel. For example, higher versions of Android strictly limit the loading of unofficial native libraries, file system access, and thread priority settings, which can directly affect the normal operation of games and drivers. Next, let’s discuss the 32-bit and 64-bit architectures. The core limitation of the 32-bit armv7a architecture is that it can only use a maximum of 4GB of virtual address space, which cannot break the addressing limit. Large memory allocations and VRAM mappings can easily trigger address overflow, and it also does not support 64-bit atomic operations and wide vector operations. Multi-threaded synchronization and the execution efficiency of rendering instructions are low. The 64-bit arm64-v8a architecture uses 48-bit addressing space, supports extremely large heap memory allocations, and complete Vulkan extension instructions. However, higher versions of Android have imposed strict permission limits on loading unofficial native libraries and thread priority settings. When a 32-bit library runs in a 64-bit environment, it will execute via a compatibility layer that incurs additional kernel switch overhead. Some new phones have directly removed support for the 32-bit instruction set, which can result in failures to load 32-bit libraries. Next, let's discuss thread scheduling and CPU affinity binding. The thread scheduling in the Android system is based on the Linux kernel's Completely Fair Scheduler (CFS). It allocates CPU time slices to threads by calculating virtual run times and controls the execution order of threads based on nice values, priorities, and cpusets. CPU affinity binding restricts specified threads to execute on certain CPU cores, avoiding the kernel scheduler from migrating threads between different cores. The underlying principle here is that in the phone's big.LITTLE architecture, big cores have high performance and stable frequency, while little cores have low frequency and weak performance. If not bound, the scheduler will automatically assign threads to big or little cores based on the load. Binding can fix critical threads, such as rendering threads, to big cores, reducing switching delays. However, in higher versions of Android, due to kernel control and SELinux restrictions, non-system applications cannot obtain complete binding permissions. Additionally, thread priority adjustments may be affected by system power consumption control, leading to binding strategies becoming ineffective.
